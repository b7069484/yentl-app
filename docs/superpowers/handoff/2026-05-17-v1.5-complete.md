## Production limitations surfaced during deploy

Two real production constraints were uncovered during the Vercel deploy QA pass. Neither was an in-scope blocker for the v1.5 deploy; both are flagged here for v1.6.

### 1. YouTube ingest blocked on Vercel cloud IPs (high impact)

**Symptom:** `POST /api/youtube-ingest` from the deployed function returns `{"error":{"code":"PRIVATE","message":"Video is private or login-required"}}` for public videos that work fine in local dev.

**Cause:** YouTube's bot detection blocks AWS Lambda IPs (Vercel's serverless runtime) using a "video is private" cover error. This is well-documented; yt-dlp issues #11868, #11744, etc. Many cloud-hosted projects hit this.

**Workaround attempted in v1.5:** `--extractor-args "youtube:player_client=mweb,tv_embedded,web"` — sometimes bypasses the block. Did NOT work for the test video on this run.

**Workarounds for v1.6 (require separate work):**
- Use a residential proxy (paid)
- Run YouTube ingest in Vercel Sandbox or a separate worker outside Vercel
- Migrate to YouTube Data API v3 + OAuth (more complex but legitimate)
- Use a third-party transcript service (RapidAPI / ScrapingBee / etc.)

**Local dev unaffected.** `pnpm dev` + paste a YouTube URL works flawlessly on a residential connection. The code path is correct; only YouTube's IP filter is the blocker.

### 2. Audio file upload limited to ~4.5MB on Vercel (medium impact)

**Symptom:** `POST /api/transcribe-batch` (multipart with a 7.4MB MP3) returns `FUNCTION_PAYLOAD_TOO_LARGE` from Vercel's gateway before the function code runs.

**Cause:** Vercel's serverless function request-body limit. The v1.3 architectural choice to drop Vercel Blob and use server-mediated multipart uploads (commit `66cb801`) made the path simpler in dev but hits this hard limit in prod. The v1.4 streaming-upload work doesn't help — the gateway rejects oversized bodies BEFORE the function executes.

**Validated working on Vercel:**
- Audio files under ~4.5MB (short clips, e.g. 1-min news segment): ✅ works end-to-end with diarization
- Text doc / mic / media URL paths: not retested in v1.5 but architecturally sound (no large-body uploads through the function)

**Fix for v1.6:** Re-add `@vercel/blob` and the client-direct upload pattern from Sprint 1. The architecture: client requests an upload URL → uploads directly to Blob → server gets the blob_url and calls Deepgram's `transcribeUrl` (which already exists at `app/api/transcribe-batch/route.ts`'s JSON branch). All the server-side pieces are in place; the client flow needs ~60 min of work.
