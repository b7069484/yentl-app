## Executive Summary

Yentl has the beginnings of a strong product flow, but the current app has two competing realities:

1. The internal flow atlas and current screenshots propose a complete product journey: public product page -> account/auth -> source choice -> source-specific ingest -> live/review workspace -> claim/marker detail -> export/library.
2. The live app route currently behaves more like a demo launcher: sparse landing page -> source picker -> session workspace, with account/auth, trust navigation, save/library, and public discovery only partially wired.

The user's core complaint is valid: there is missing connective tissue, especially from discovery/acquisition to account value, from account value to saved library, and from some source-specific intake screens into the final workspace. However, the specific "why are session tabs visible before a session exists?" issue is mostly handled correctly in the current live app. The session header/tabs do not appear on source-picker and source-ingest screens. They appear after a started or loaded workspace, where they belong.

The highest-risk PM issues are:

- P0: Auth routes exist but break locally when Clerk env is absent, while the product flow atlas says auth/account is part of the journey.
- P0: Privacy/trust pages say no user accounts and no session histories, while the app has Clerk auth routes and a saved sessions library.
- P1: The live public landing page is not the same as the fuller promo/product-page screenshot. It does not explain acquisition, account value, trust, pricing, or source options at the standard expected from a major product.
- P1: The source picker is visually clean, but it still leaves the user choosing among six modes before the app fully explains which path is best for their current situation.
- P1: Microphone pre-start screen has no visible "Back to sources" escape, unlike other source panes.
- P1: Live Watch can show a claim in the evidence queue while the session nav says "Claims 0" and the Claims page says "No matches."
- P2: Media URL pre-ingest has a disabled "Process" button whose label is visually missing or nearly invisible in the current render.
- P2: Trust/contact links include missing routes: /contact and /docs/engagement-gate return 404.

## Major-App Flow Benchmark

For a product like this, users expect a flow similar to mature media/productivity tools:

1. Public discovery
   - Product promise, who it is for, concrete examples, trust posture, privacy, pricing, and one clear primary CTA.
2. Acquisition/account
   - Try demo or sign in/sign up, with a clear reason for an account: save history, sync devices, export records, manage privacy.
3. Signed-in home/portal/dashboard
   - Recent work, create/import/start options, saved sessions, status, and clear next action.
4. Source selection/import
   - A guided intake based on user job, not just technical input type.
5. Processing/waiting
   - Clear status, ETA or stage, fallback routes, and cancel/change-source paths.
6. Core workspace
   - The source content remains the anchor; transcript, claims, markers, sources, and controls are arranged around what the user came to inspect.
7. Review/detail
   - Claims and markers have drill-downs with evidence, transcript context, and navigation back to source moment.
8. Durable outcome
   - Save, export, share/report, library restore, and account/sync state are understandable.

Yentl's internal /project/flows atlas describes most of this. The live app only partially implements it.

## Product Journey Audit

### 1. Public Discovery: Live Root Page

Route: /
Source: app/page.tsx
Screenshot contrast: public/visual-evidence/flow-screenshots/current/promo-product-page.png

What works:

- The live root is clean and immediately communicates "live fact-checker."
- The visual hierarchy is calm and not cluttered.
- The primary CTA "Start a session" is obvious.

What is missing:

- No sign in / sign up entry point, despite auth routes existing and the internal flow atlas treating account as part of the journey.
- No trust page links, privacy, methodology, examples, pricing, or "how it works" navigation in the live root.
- The landing copy says "Multi-speaker - English - Browser mic" even though the source picker supports browser tab, text, audio/video upload, YouTube, and media URL. This undersells the actual app and steers mental model back to mic-only.
- The richer promo/product-page screenshot shows a much more complete acquisition page: yentl.it brand, nav, "How it works," "Features," "Pricing," "About," "Try the live demo," and product proof. That is the right product direction, but it is not the live root.

PM verdict:

This is the first major connective-tissue gap. The current live landing page is a demo entry, not a product acquisition flow.

Recommendation:

- Decide whether / is a product marketing page or a direct demo launcher.
- If / is public discovery, use the richer product-page direction.
- If / is demo-first, add clear secondary routes: "How it works," "Privacy," "Methodology," "Sign in," and "Saved sessions."

### 2. Auth / Account

Routes: /signin, /signup
Sources: app/signin/[[...rest]]/page.tsx, app/signup/[[...rest]]/page.tsx, app/layout.tsx

What works:

- Clerk routes are present in file structure.
- The internal atlas correctly identifies auth as a product-flow requirement.

What fails:

- In local dev without NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY, RootLayout intentionally skips ClerkProvider, but /signin and /signup still render Clerk SignIn/SignUp components. Result: both routes throw "useSession can only be used within the ClerkProvider" locally.
- The live root does not link to /signin or /signup.
- Privacy copy says Yentl does not maintain user accounts or session histories, which conflicts with auth route existence and the saved sessions product direction.

PM verdict:

Auth is currently neither absent nor productized. It is a broken side-route plus a conceptual promise in the atlas. That is worse than choosing one clear mode.

Recommendation:

- Make account strategy explicit:
  - "Guest mode with local sessions only" for current v1, or
  - "Account mode with synced sessions" if Clerk is now in scope.
- Until account is truly ready, hide or guard /signin and /signup gracefully.
- Update privacy and onboarding copy to distinguish local IndexedDB saves from server-side account sync.

### 3. Signed-in Home / Portal / Dashboard

Route: /sessions
Source: app/sessions/page.tsx

What works:

- Saved sessions have a dedicated library.

### 4. Source Selection

Route: /session before start
Source: components/session/source-picker.tsx

What works:

What is weak:

PM verdict:

Good direction, but it needs one more layer of guided decision support: "Best for..." routes, clear extension prerequisites, and source-specific consent.

Recommendation:

- Keep the pre-session surface quiet.
- Replace the broad consent banner with source-specific consent gates:
  - Mic: people around you and local law.
  - Browser tab: tab audio capture and extension permission.
  - Upload: file handling and processor disclosure.
  - Text: no recording copy needed.
- Add a small "Not sure? Start with any video/page" explanation under the featured path.

## Core Session Workspace Audit

### Session Shell / Header

Source: components/session/session-shell.tsx

What works:

- The session header appears only when startedAt exists. This is the right boundary.
- Header controls are appropriate during a session: Library, status pill, Overview, Watch, Transcript, Claims, Markers, Sources, Save, Export, End.
- Watch tab appears only for playable sources.

What is questionable:

- "Sources" can be a correct control, but it is semantically odd as a destructive/reset path in the session header. It can mean "view evidence sources" or "choose another input source." In this app it means choose another input source. That ambiguity matters in a fact-checking product where "sources" also means citations.
- Export is visible even before there is meaningful output in some states.
- In screenshots, the consent reminder can appear inside the session workspace after header, which competes with the content if no live recording is happening.

Recommendation:

- Rename header "Sources" to "Change source" or "New source" if it resets/changes input.
- Reserve "Sources" for evidence/citations.
- Make consent contextual to capture state, not global page state.

## Final PM Assessment

The app is not overloaded with top navigation during media/content selection, which was the user's most specific screen-level concern. That boundary is mostly right. The larger issue is journey architecture: Yentl has strong individual surfaces, but the live app does not yet feel like a complete major-app flow from discovery to account value to home/dashboard to source selection to durable output.

The best product direction is already visible in the internal atlas and extension screenshot: same-page analysis, source-aware intake, Watch as the media-centered review surface, and library/export as the durable end state. The next work should not be another broad redesign. It should reconcile the live app with that intended flow, remove internal/dev surfaces from end-user routes, and make every state tell the truth about where the user is: pre-session, importing, waiting, live, replay/review, saved, or exported.
