# Yentl Loop Issue Ledger

Last reconciled: 2026-06-09T03:15:57-04:00

This ledger prevents recurring automation runs from rediscovering the same issue without resolving it. Every audit finding must be deduped here before it is treated as new.

## Status Values

| Status | Meaning |
|---|---|
| `new` | Newly observed; not yet triaged. |
| `ready_for_fix` | Narrow, reproducible, and safe for the fix loop. |
| `fixed_pending_verify` | Code or docs changed; watchdog/audit must verify in a later run. |
| `verified_fixed` | Later audit confirmed the issue is gone. |
| `reopened` | Same issue recurred after a fix. |
| `blocked_needs_context` | Needs human/product decision. |
| `blocked_needs_verification` | Fix may exist but required proof could not run. |
| `deferred` | Intentionally postponed with rationale. |
| `escalated` | Repeated or high-risk issue requiring Israel's attention. |

## Rules

- Search this table before adding a row.
- Same route plus same symptom is the same issue unless the root cause is clearly different.
- A fix loop may select only one `ready_for_fix` row per run.
- Privacy, legal, terms, subprocessors, and other human-approval-gated public claims must not be marked `ready_for_fix` for the normal `ui-mobile-fix` loop unless a human approval note explicitly permits it.
- A row cannot become `verified_fixed` in the same run that changed code unless an independent verification command/screenshot proves it.
- After three recurrences or three no-progress runs, mark `escalated`.
