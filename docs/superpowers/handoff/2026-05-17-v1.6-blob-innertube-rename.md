## Task-by-task

### v1.6-A — Vercel Blob client-direct upload (commit `5ceb54a`)

- `pnpm add @vercel/blob@^2.3.3`
- `/app/api/upload-audio/route.ts` — `handleUpload` route. Validates content types (audio/* only), enforces 500MB max, adds random suffix for collision safety.
- `lib/client/audio-ingest.ts` — new `transcribeViaBlob()` helper. Branches on `file.size >= BLOB_UPLOAD_THRESHOLD_BYTES (4MB)` — large files go through Blob, small files keep the multipart path (no token needed in dev).
- `components/session/ingest-panes/audio-ingest-pane.tsx` — new `"uploading"` phase with progress bar for large files. Small files skip straight to "Transcribing…".
- 17 new tests (9 for the route, 6 for the client path, 2 for the pane).

**Architecture:** Client requests an upload URL → uploads DIRECTLY to Vercel Blob (bypasses our function) → server gets back the blob URL → client calls `/api/transcribe-batch` with `{ blob_url, duration_sec }` (already-implemented JSON branch) → Deepgram fetches the URL directly via `transcribeUrl` mode. No body ever flows through our function for large files.

**Server-side multipart path NOT removed** — it stays for small files (no Blob token required, dev-friendly).

  - Existing yt-dlp path renamed to `fetchViaYtDlp(videoId)` — unchanged in logic

- 25 new tests across `tests/youtube-innertube.test.ts` (20) and updates to `tests/youtube-captions.test.ts` (5 orchestration tests)

**Verified state:**
- **Local dev:** Innertube successfully fetches captions for public videos. yt-dlp fallback also works on residential IPs.
- **Vercel production:** Both Innertube and yt-dlp blocked by the same cloud-IP filter. POST `/api/youtube-ingest` returns `{"error":{"code":"PRIVATE","message":"Video is private or login-required"}}` for public videos.

This confirms the block is at the IP/network layer — YouTube doesn't care which API the request hits if it's coming from AWS Lambda IP space. Innertube reduces our vulnerability to yt-dlp upstream breakage (it's now the primary path everywhere it works), but doesn't solve the cloud-IP problem.

## Required user actions

Three actions, in order of urgency:

### Action 1 — yentl.it DNS (5-min fix at GoDaddy)

The domain is reserved at GoDaddy. Vercel has added it to the project and aliased it to the production deployment. The only remaining step is pointing the DNS records at Vercel.

**Option A (recommended) — Set an A record:**

1. Log into GoDaddy → My Products → DNS for `yentl.it`
2. Add a record:
   - **Type:** `A`
   - **Host (Name):** `@` (this means the root domain `yentl.it`, not a subdomain)
   - **Value:** `76.76.21.21` (Vercel's apex IP)
   - **TTL:** `1 Hour` (or default)
3. **If GoDaddy already has a "Parking" A record on `@`, replace its value with `76.76.21.21`. Don't leave both.**
4. Save. Propagation takes 5–30 minutes typically; up to 24h worst case.
5. **Verify** with `dig yentl.it` or `curl -I https://yentl.it/` — should return Vercel's response.

For the `www.yentl.it` variant (optional):
   - **Type:** `CNAME`
   - **Host (Name):** `www`
   - **Value:** `cname.vercel-dns.com`

**Option B (alternative) — Change nameservers:**

1. GoDaddy → DNS for `yentl.it` → **Nameservers** → Change → use these:
   - `ns1.vercel-dns.com`
   - `ns2.vercel-dns.com`
2. This gives Vercel full DNS control of the domain (cleaner for advanced setups). Don't use this if you want to keep GoDaddy email/other records on the domain.

After either option, Vercel auto-verifies within minutes and starts serving https://yentl.it.

### Action 2 — Provision a Vercel Blob store (2-min setup)

Required for audio uploads ≥ 4MB. Without it, the `/api/upload-audio` route returns 400 and the audio-ingest-pane falls back to multipart only.

1. Vercel dashboard → **yentl** project → **Storage** tab → **Create Database** → **Blob**
2. Give the store a name (e.g., "yentl-audio")
3. Vercel auto-creates the env var `BLOB_READ_WRITE_TOKEN` in Production + Preview + Development
4. Pull it locally: `vercel env pull` (writes to `.env.local`)

That's it. The route automatically picks up the token on next request. No redeploy required.

### Action 3 — YouTube fix (medium effort, optional for now)

YouTube ingest is blocked on Vercel cloud IPs. Pick one of these paths when you want YouTube to work in production:

**Option Y1 (cheapest, ~$5/mo) — Tiny self-hosted worker:**
- Spin up a $5/mo Hetzner / DigitalOcean / Linode VPS (residential-friendly IPs)
- Install yt-dlp + a minimal Node server with a single `/captions` endpoint
- Point our `fetchViaYtDlp` at that endpoint via an env var (e.g., `YT_DLP_WORKER_URL`)
- Pros: Reliable, cheap, you own the infra. Cons: Maintenance overhead.

**Option Y2 (per-call cost) — Third-party transcript API:**
- Services like SearchAPI.io YouTube Transcripts (~$0.001/call, free tier 100/mo), Apify, or RapidAPI's various transcript scrapers
- Add a `fetchViaTranscriptAPI(videoId)` path that hits the service
- Pros: Zero infra. Cons: Per-call cost, external dependency, vendor risk.

**Option Y3 (most expensive, most reliable) — Residential proxy:**
- Services like Smartproxy or BrightData (~$15–50/mo entry tier)
- Set `HTTPS_PROXY` env var on Vercel pointing at the proxy endpoint
- Innertube + yt-dlp both work through it
- Pros: Drop-in fix, no code change. Cons: Ongoing cost, can be flagged by YouTube too.

**Option Y4 (free, fragile) — Cloudflare Workers proxy:**
- Deploy a small CF Worker that fetches `https://www.youtube.com/watch?v=…` and proxies caption requests
- Cloudflare IPs are different from AWS Lambda IPs; YouTube might let them through
- Pros: Free tier. Cons: Cloudflare may also be on YouTube's blocklist; needs prototyping.

**My recommendation:** Y1 (the $5/mo worker) is the most pragmatic if you want this fixed permanently with no ongoing per-call cost. Y4 first to see if Cloudflare works gives you a free option to try.

   - Reduces function bundle size by ~25MB
