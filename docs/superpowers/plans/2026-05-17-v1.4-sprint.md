### Task A — Synthesis persistence in saved sessions

**Why:** v1.3 known issue #6. `restoreSession` clears `synthesis` to null. For a session built from a 2-hour podcast, the synthesis paragraph + per-speaker verdicts are the most expensive AI output and the user expects them back on restore.

**Files:**
- Modify: `lib/types.ts:67-76` (add optional `synthesis` to `Session`)
- Modify: `lib/client/session-store.ts:192-206` (`restoreSession`), `:208-220` (`toSession`)
- Create: `tests/synthesis-persistence.test.ts`

**Contract decision:** Persist only `"fresh"` and `"refreshing"` synthesis states (the ones with valid text). Drop `"warming"` and `"error"` on save (the latter has no actionable text). On restore, downgrade `"refreshing"` to `"fresh"` since the refresh job won't be running.

**Steps:**

- [ ] Write failing test asserting that `toSession()` includes synthesis when state is "fresh"
- [ ] Write failing test asserting `restoreSession()` populates `synthesis` field with the saved value
- [ ] Write failing test asserting "warming"/"error" states are not persisted (synthesis is undefined in the saved Session)
- [ ] Run tests → expect FAIL
- [ ] Add `synthesis?: PersistedSynthesis` to `Session` type in `lib/types.ts`. Define `PersistedSynthesis` as `{ text: string; headlines: string[]; per_speaker_verdicts?: SpeakerVerdict[]; at: number }` — strictly the persistable shape (no state machine).
- [ ] Modify `toSession()`: if `s.synthesis?.state === "fresh"` or `"refreshing"`, attach `{ text, headlines, per_speaker_verdicts, at }` to the returned Session
- [ ] Modify `restoreSession(session)`: if `session.synthesis` is present, set store synthesis to `{ state: "fresh", ...session.synthesis }` (downgrade refreshing→fresh)
- [ ] Run tests → PASS
- [ ] Run full test suite (`pnpm -C "..." test:run`) — confirm 940+ tests still green
- [ ] Commit: `feat(persistence): preserve synthesis across save/restore`

**Acceptance:** Save a session with fresh synthesis → reload → restore → SynthesisCard renders the previous text + per-speaker verdicts. Confirmed via unit test; manual UI check in Task I.

### Task C — Streaming audio upload for files >50MB

**Why:** v1.3 known issue #2. Current `transcribe-batch` reads the entire `File` into a `Buffer` in memory — fine for ≤50MB, risky toward the 500MB cap (Vercel function memory limit).

**Files:**
- Modify: `app/api/transcribe-batch/route.ts:57` — branch on file.size
- Create: `lib/client/audio-upload.ts` — adaptive uploader
- Modify: `components/session/ingest-panes/audio-file-pane.tsx` (or wherever the audio pane lives) — use the new uploader
- Create: `tests/streaming-upload.test.ts`

**Contract decision (largest-impact):** For files >50MB, **stream** the multipart body directly into Deepgram's transcribeFile using `Request.body` as a `ReadableStream` → convert to `Readable` → pipe. The Deepgram SDK v5 `transcribeFile` accepts a Buffer; check if it accepts a Readable (likely yes — Node typically). If not, fall back to streaming-to-tempfile and reading the tempfile.

**Steps:**

- [ ] Read Deepgram SDK v5 source/docs in `node_modules/@deepgram/sdk` to confirm transcribeFile signature accepts a Readable stream
- [ ] Write failing test: a 60MB mock file is uploaded → multipart path → confirms no `Buffer.from(arrayBuffer)` call (memory-buffering check via spy)
- [ ] Write failing test: a 1MB mock file → still uses Buffer path (small files unchanged)
- [ ] Run tests → FAIL
- [ ] In route.ts multipart branch: if file.size > 50 * 1024 * 1024, stream the file via `file.stream()` → pipe through Web→Node stream conversion → pass to Deepgram
- [ ] If Deepgram SDK can't accept a stream, write to `tmpfile()` via `pipeline` and pass a `createReadStream` of the tempfile
- [ ] Add cleanup: delete tempfile in `finally` block
- [ ] Run tests → PASS
- [ ] Commit: `feat(audio): stream large file uploads (>50MB) to Deepgram`

**Acceptance:** A 100MB synthetic audio file (or mocked) upload does not allocate a 100MB buffer; tests assert via stream-based assertions. Live verification in Task H if budget allows.

### Task D — yt-dlp production packaging

**Why:** v1.3 known issue #1. `yt-dlp` is on PATH locally but not on Vercel's serverless Node runtime. Without this, every YouTube ingest fails in production with `YT_DLP_MISSING`.

**Files:**
- Create: `lib/server/yt-dlp-binary.ts` — resolves the binary path
- Modify: `lib/server/yt-dlp-runner.ts` — accept a path argument
- Modify: `lib/server/youtube-captions.ts:251` — call `getYtDlpBinaryPath()` and pass it
- Modify: `package.json` — add `yt-dlp-wrap` OR `youtube-dl-exec` (decision in step 1)
- Create: `vercel.json` if not present — set `functions.{"app/api/youtube-ingest/route.ts": {"includeFiles": "bin/**"}}`
- Create: `tests/yt-dlp-binary.test.ts`

**Investigation (do this first):** Read the official Vercel Functions docs at https://vercel.com/docs/functions/runtimes to confirm:
1. Whether Vercel Node functions can spawn child processes (yes — Node runtime)
2. Whether bundled binaries via `includeFiles` get marked executable on extraction (usually requires `chmod +x` at runtime)
3. The Lambda layer alternative (probably overkill)

**Recommended approach:** Use [`yt-dlp-wrap`](https://www.npmjs.com/package/yt-dlp-wrap) — it auto-downloads the binary at install time OR runtime. For Vercel: download at install via `postinstall` script, store under `node_modules/yt-dlp-wrap/bin/`, then reference that path at runtime. Or use `youtube-dl-exec` which bundles a binary directly.

**Steps:**

- [ ] Investigate the two npm options and pick one — preference: `yt-dlp-wrap` (auto-update at install, lighter on repo)
- [ ] `pnpm -C "..." add yt-dlp-wrap`
- [ ] Write failing unit test: `getYtDlpBinaryPath()` returns a string ending in `yt-dlp` (or `yt-dlp.exe` on Windows for dev)
- [ ] Write `lib/server/yt-dlp-binary.ts`: try `YT_DLP_PATH` env, then `yt-dlp-wrap`'s bundled binary, then PATH lookup ("yt-dlp")
- [ ] Modify `lib/server/yt-dlp-runner.ts` to export a `spawnAt(path, args)` variant; keep the existing `spawn` re-export for back-compat
- [ ] Modify `lib/server/youtube-captions.ts:spawnYtDlp` to call `spawnAt(getYtDlpBinaryPath(), args)`
- [ ] Run tests → PASS
- [ ] `pnpm -C "..." test:run` — confirm all YouTube caption tests still green (they should: mocks are at yt-dlp-runner)
- [ ] Document Vercel deploy steps in the v1.4 handoff: `vercel.json` `includeFiles` glob, `chmod` step if needed, env var
- [ ] **Do NOT run `vercel deploy`.** Stop short of deployment. The user authorizes deploy separately.

### Task E — Reassign menu mid-block split

**Why:** v1.3 known issue #4. Today `reassignUtterance(index, speaker)` swaps the WHOLE segment's speaker. When Deepgram bundles two sentences from different speakers into one utterance, the user can't fix it without splitting the block.

**Files:**
- Modify: `lib/client/session-store.ts` — add `splitSegmentAt(index, splitTime, newSpeakerId)` action
- Modify: `components/session/reassign-speaker-menu.tsx` — add "Split & reassign…" UI
- Create: `tests/reassign-split.test.ts` — store action tests

**Contract decision:** Split at the **time** of a word boundary, not a character offset — the Deepgram word-level timestamps in the original payload aren't preserved at TranscriptSegment level, so we'll split at the boundary between words by text-position-fraction → time-fraction. Acceptable approximation. Cascading: claims/markers strictly inside the post-split portion move to the new segment; the rest stay.

Goodbye now." }` produces two segments — original truncated to `{ start: 10, end: splitTime, text: "Hello world." }` and new segment `{ start: splitTime, end: 20, text: "Goodbye now.", speaker_id: 1 }`
- [ ] Write failing test: claims with `utterance_start >= splitTime && utterance_start <= original_end` get `speaker_id: newSpeakerId`
- [ ] Write failing test: markers fully within the post-split portion get the new speaker; markers straddling the split are unchanged
- [ ] Write failing test: out-of-bounds index → no-op
- [ ] Run tests → FAIL
- [ ] Implement `splitSegmentAt` in session-store: clone state, mutate transcript[index].end, insert new segment after at index+1, cascade claims/markers, register speaker
- [ ] Run tests → PASS

- [ ] After picking a word, ask which speaker the *post-split* portion belongs to (same speaker list)

- [ ] Run tests → PASS
